MuJoCo HalfCheetah · Heuristic Learning · Runtime Intelligence

一只经典"小猎豹"，还能有什么新故事？

MuJoCo HalfCheetah 是具身智能、强化学习和连续控制里的经典任务。一个成熟到不能再成熟的 benchmark，似乎已经很难讲出新东西。

但翁家昱的工作提醒我们：创新未必只发生在更大的模型和更重的训练里。可解释步态、姿态规则、在线规划，再加上 AI coding agent 的持续迭代，也能把启发式系统推到接近常见 Deep RL 的结果量级。

Genesis-HL 选择了另一条更小的路：不追高分，不比规模，而是把这个过程拆开，让我们看见一个智能系统如何在运行层中观察反馈、质疑假设、重构搜索空间，并逐步长出更好的行动结构。

核心命题
世界模型负责预测，元思考负责纠偏，智能运行层负责让纠偏持续发生。
CPG：先给小猎豹一个稳定节拍，让它有基础步态。
MPC：再让它在行动前短暂预演未来，学会"边跑边想"。
Genesis-HL：把反馈、诊断、重构和验证组织成可审计运行闭环。

先说清楚：CPG 和 MPC 到底是什么？

这两个词听起来技术，但背后的直觉很简单：一个负责"有节奏地动起来"，另一个负责"边走边试想未来几步"。

C

CPG（Central Pattern Generator，中央模式发生器）：身体里的节拍器

CPG 可以理解成一种周期步态生成器。它不需要神经网络训练，而是用类似"节奏"的方式控制关节：什么时候抬腿，什么时候发力，前后腿如何错开。

直觉：像人走路时的基础步伐。它不聪明，但稳定、便宜、可解释。
M

MPC（Model Predictive Control，模型预测控制）：短视窗里的小型世界模型

MPC 会在真正行动前，先用模型或仿真器预测未来几步：如果接下来几步稍微这样修正，会不会更往前？会不会更稳？然后选择当前最合适的一步执行。

直觉：像一边跑一边快速预演未来几步。它更聪明，但更耗计算。
在这个 demo 里，CPG 是基础行动层；MPC 是短视窗预测层。二者合起来，就是"先有稳定步态，再边跑边想"。

为什么这不是普通机器人 demo？

Reframe

真正的主角不是小猎豹，而是运行层。

HalfCheetah 只是一个舞台。它让我们能清楚看到：动作有没有让身体前进？失败来自力度、步态结构，还是预测层不够好？

传统叙事会问：模型分数多高？Genesis-HL 更关心：系统为什么这样改？反馈如何改变下一轮搜索空间？什么时候该停止调参、转而接入新结构？

机器人在跑，真正被验证的是智能运行层。

两个新词：元思考与智能运行层

为了避免把这个 demo 误解成普通控制算法组合，我们先把两个关键词说清楚： 元思考不是"想得更久"，智能运行层也不是某个单独模型。

M

元思考：判断"该怎么改问题"

元思考不是多生成几步推理，而是当反馈出现时，系统能反过来检查自己的假设： 这次失败是因为参数不对、结构不对，还是问题定义本身错了？

在小猎豹项目里： 放大动作失败后，我们没有继续加大力度，而是判断"问题不在力度，而在步态结构"，于是转向 gait search。
R

智能运行层：让改进持续发生的组织层

智能运行层不是 CPG、MPC 或某个神经网络，而是把策略、环境、日志、反馈、诊断、搜索和验证组织起来的运行结构。

在小猎豹项目里： run_eval、trials.jsonl、summary.csv、failure_hint、参数扫描、多 seed 回归、CPG vs MPC 对照，共同构成了这个运行层。
简单说：世界模型负责预测未来，元思考负责判断预测和策略是否需要纠偏，智能运行层负责把这种纠偏变成可重复、可审计的工程闭环。

世界模型、元思考、智能运行层如何闭环？

如果只看单个动作，这只是控制问题；如果看整个循环，它就是智能如何生长的问题。

01

行动策略

CPG 先给出一个稳定、可解释的基础步态。

02

世界预测

MPC 用 MuJoCo 在短视窗里预演候选动作。

03

现实反馈

return、位移、动作幅度、耗时被记录下来。

04

元思考

判断失败说明什么：调参数、改 gait，还是加新层？

05

运行层重构

把下一轮搜索、回归验证、干净对照组织起来。

把它画成一张运行层框架图

如果说上面的五个节点解释了"智能如何闭环"，那么下面这张图展示的是： 这个闭环在小猎豹项目里具体长成了什么样。 CPG 负责基础步态，MPC 负责短视窗预测，MuJoCo 提供世界反馈； 运行日志、元思考和搜索空间重构，则把每一轮失败和改进变成下一轮行动。

这张图的重点不是某一个算法，而是层与层之间的关系： CPG 让它先能跑，MPC 让它边跑边想，元思考让系统知道为什么要改， 智能运行层让这些改动持续发生。

故事怎样发生：从乱动，到边跑边想

v0.1

先让它留下痕迹

random 和初始 CPG 都不强，但日志系统建立起来了。没有日志，就没有反馈；没有反馈，就没有运行层学习。

v0.2

第一次失败改变了问题

放大动作幅度没有变好，反而更差。系统得到第一条重要反馈：不是力度问题，是 gait 结构问题。

v0.3

找到 alternating gait

从 rear drive、front drive、bound 等步态中，alternating 第一次产生明确正向位移。搜索空间开始收缩。

v0.3.4

稳定 base gait 出现

多 seed 回归确认 CPG/PD base gait 稳定正回报。系统不再只是偶然成功，而是得到可复现结构。

v0.4

接入 residual MPC

当 CPG 微调进入平台期，我们不再继续拧参数，而是在稳定步态上接入短视窗规划。

v0.4.6

形成干净对照

同一个环境、同一个 seed，比较 CPG 和 MPC：MPC 在短中程中带来更高 return 和更大位移。

Clean comparison：CPG vs MPC

这不是为了证明 MPC 天下无敌，而是为了证明：在稳定 base gait 上增加一个短视窗预测层，确实能产生可测量增益。

三条路线怎样互补？

传统 RL

用大量环境交互和 reward 优化训练出强 policy。

  • 强在性能上限
  • 弱在解释和研发透明度
  • 适合最后冲高性能
+

翁家昱 HL

用 coding agent 维护启发式系统，让规则、测试、日志、回放持续生长。

  • 强在工程迭代速度
  • 让 heuristic system 重新有生命力
  • 适合把复杂系统快速推高
Genesis-HL

它补上的，是运行层可见性。

Genesis-HL 不追最高分，而是把每一轮为什么失败、为什么转向、为什么接入新层讲清楚。

它像一个研发驾驶舱：让 LLM 团队、机器人团队和管理层都能看见智能系统如何从反馈中改进。

商业上它说明什么？

这个 demo 的商业价值不是"我有一个机器人控制器"，而是"我有一种低成本验证智能系统运行闭环的方法"。

降低试错成本

先用小 demo 看清问题在哪里，而不是直接进入大规模训练或重工程。

降低解释成本

把结果变成日志、指标、对照和故事，让不同团队能在同一张图上讨论。

降低决策风险

每一轮投入前，都知道上一轮反馈如何改变下一步方向。

所以，这只小猎豹真正展示了什么？

它展示的不是 SOTA 控制能力，而是一种智能运行层的早期形态：可观察、可反馈、可诊断、可收缩、可验证、可分层、可对照。

世界模型让它在行动前试想未来；元思考让我们根据反馈质疑原来的假设；智能运行层把这些预测、行动、反馈和修正组织成可持续的研发闭环。

HalfCheetah 是舞台，CPG/MPC 是角色，真正被验证的是：智能可以在运行层中生长。
